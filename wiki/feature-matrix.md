# Open Arena feature gap matrix

This matrix summarizes the current implementation against the target stack described in [`notes.md`](./notes.md). The priorities are intended to help sequence the work: **High** means the feature closes an architectural gap in the near-term product, **Medium** means it materially improves platform breadth after the core is stable, and **Low** means it is valuable but should follow the more foundational modules.

Status legend:

- **Working**: implemented and usable in the current repository.
- **Partial**: some groundwork exists, but the feature is incomplete or not yet platform-grade.
- **Not implemented**: no meaningful implementation in the current repository.

|---|---|---|---|---|---|---|

| [Dataset and benchmark registry](./notes.md#1-dataset-and-benchmark-registry) | Version-aware dataset tracking | Partial | Some providers expose version fields such as HF revision, Phoenix version, Braintrust version, and Weave version. | Version handling is provider-specific metadata, not a unified registry capability. | High | Introduce a normalized dataset identity and version schema that every run can reference. |

| [API and integration layer](./notes.md#7-api-and-integration-layer) | Integration surface for remote orchestration | Not implemented | No dedicated integration endpoints or service layer exist. | The platform cannot yet be embedded cleanly in broader evaluation workflows. | Low | Layer this on top of the API module once core entities are stable. |

## Priority summary

The highest-priority gaps are the ones that turn the current repository from a strong CLI runner into a platform: a normalized benchmark registry, a portable run and leaderboard artifact model, step-aware execution and evaluation, and a dedicated agent-evaluation module. Those changes build directly on top of the parts that already work well today.

The medium-priority group expands breadth once that foundation is in place: richer verifier engines, synthetic-data pipelines, API exposure, and protocol-specific agent integrations. Low-priority work, especially sandboxing and deeper orchestration integrations, should come after the domain model is settled so those features do not force premature abstractions.
